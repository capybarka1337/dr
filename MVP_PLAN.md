# План MVP приложения «Поздравления с днем рождения»

## 1. Видение и цели продукта
- **Цель MVP**: предоставить пользователю простой способ создать персональное поздравление и моментально поделиться им по ссылке, чтобы получить обратную связь о востребованности продукта.
- **Основная ценность**: готовые шаблоны + возможность кастомизации текста и визуальной темы без регистрации.
- **Критерий успеха MVP**: пользователи создают и отправляют поздравления спустя ≤2 минуты после входа на сайт, возвращаются за повторным использованием и рекомендуют сервис знакомым.

## 2. Целевая аудитория и ключевые сценарии
### Сегменты аудитории
1. **Молодые профессионалы 20–35 лет** — хотят быстро поздороваться с друзьями/коллегами. 
2. **Родители и родственники** — ищут готовые тёплые тексты для близких.
3. **Люди, живущие на расстоянии** — ценят возможность отправить красивую открытку онлайн.

### Основные сценарии использования
1. Пользователь выбирает готовое поздравление → копирует ссылку → отправляет в мессенджер.
2. Пользователь настраивает текст и визуальную тему → просматривает предпросмотр → отправляет ссылку.
3. Получатель переходит по ссылке → видит персонализированную открытку → при желании делится дальше.

## 3. Функциональность MVP
### Must Have (обязательно)
- Landing с кратким описанием ценности и CTA «Создать поздравление».
- Блок «Готовые поздравления» (2–3 шаблона с мгновенной генерацией ссылки).
- Форма создания кастомного текста (имя получателя, обращение, основной текст, подпись).
- Выбор визуальной темы (существующие градиенты: aurora, sunrise, lagoon).
- Страница предпросмотра с адаптивным дизайном.
- Кнопка «Поделиться» с поддержкой `navigator.share` и fallback «Скопировать ссылку».
- Конструирование ссылки с параметрами в URL (без сервера).
- Русская локализация интерфейса и текстов.

### Should Have (желательно, если останется время)
- Мини-история последних поздравлений в LocalStorage.
- Возможность загрузки пользовательского emoji/тикера.
- Подсказки по тону поздравления (например, «официальное», «дружеское»).

### Won’t Have (не попадает в MVP)
- Авторизация/личный кабинет.
- Push-уведомления, рассылки.
- Платёжные сценарии и премиум-контент.
- Полноценная аналитика поведения пользователей, за исключением базовых метрик.

## 4. UX/UI и визуальный дизайн
- **Тон общения**: дружелюбный, вдохновляющий, без сложной терминологии.
- **Типографика**: акцентный заголовок (например, «Playfair Display»), читабельный текст («Inter» или «Source Sans»), размеры 32/24/18/16/14рх.
- **Цветовая палитра**: использовать существующие градиентные темы, добавить нейтральный фон (#f8fafc) и тёмный текст (#1f2933).
- **Компонентный подход**: карточки шаблонов, форма, предпросмотр, кнопка шаринга, модальное окно подтверждения копирования.
- **Респонсивность**: мобильный-first, контроль ширины формы (max 480px), оптимизация карточек в одну колонку на <768px.
- **Доступность**: читаемые контрасты, `aria-label` для кнопок, фокус-стили, поддержка клавиатуры.
- **Анимации**: лёгкие переходы (200–250 мс) при смене тем и появлении уведомлений, без перегрузки.

## 5. Работоспособность и нефункциональные требования
- **Производительность**: загрузка ≤2 сек на 4G, размер бандла <150 КБ (gzipped), lazy-loading не критичных компонентов.
- **Надёжность**: приложение полностью клиентское, без внешних зависимостей; предусмотреть graceful degradation при отсутствии `navigator.share`.
- **Безопасность**: защита от XSS через экранирование пользовательского ввода, ограничение длины полей.
- **Доступность (a11y)**: WCAG AA для основных элементов, семантическая верстка.
- **Готовность к масштабированию**: кодовая база на TypeScript, возможность добавления новых тем/шаблонов без переписывания логики.

## 6. Техническая архитектура
- **Фронтенд**: Vite + React + TypeScript (существующий стек проекта).
- **Стейт-менеджмент**: локальное состояние компонентов + Context API для тем/настроек.
- **Хранение данных**: параметры поздравления в URL (`URLSearchParams`), временные данные — в `LocalStorage` (по необходимости).
- **Модули**:
  - `App` (роутинг между Landing, Конструктором и Просмотром).
  - `DefaultGreetings` (список шаблонов).
  - `CustomGreetingForm` (форма ввода).
  - `GreetingView` (предпросмотр/страница получения).
  - `ShareButton` (логика шаринга и уведомления).
- **Стили**: CSS-модули или глобальные стили c использованием переменных, тема — набор градиентов.
- **Интеграции**: на этапе MVP нет внешних API. Для аналитики — лёгкий скрипт (например, Plausible) при необходимости.

## 8. Метрики успеха
- **Активные пользователи**: ≥100 уникальных в первые 2 недели.
- **Созданные поздравления**: ≥60% посетителей создают/просматривают открытку.
- **Время до первой открытки**: ≤120 секунд.
- **Степень завершения**: ≥80% начавших заполнять форму доходят до предпросмотра.
- **NPS/отзывы**: средняя оценка удовлетворенности ≥8/10.

## 9. Риски и планы смягчения
- **Отсутствие отклика аудитории** → запланировать пользовательские интервью после релиза.
- **Перегруженность интерфейса** → юзабилити-тесты на ранних прототипах.
- **Технические сложности с шарингом** → fallback с копированием ссылки и подсказкой.
- **Сложность текстов** → подготовить шаблоны разного тона и краткости.
- **Отсутствие аналитики** → подключить лёгкий сбор базовых метрик (просмотры, нажатия CTA).
